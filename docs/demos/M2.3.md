# M2.3 — Integrated Accessibility Closed-Loop Acceptance

## Goal

Verify the DEBUG-only typed flow:

```text
command → live snapshot → LM-first decision/fallback → validated action
        → accessibility-event debounce → fresh snapshot → verification
```

The normal STT/RuleBasedReplyEngine voice path is intentionally unchanged until M3.1. Android
accepting `performAction` is not success; only `VerificationResult.Verified` is success.

## Automated gate — no device or credential

```sh
./gradlew testDebugUnitTest assembleDebug lintDebug assembleRelease
```

Expected: all JVM tests pass, including virtual-time event debounce/timeout/cancellation and fake
read → action → read ordering. LM-disabled and provider-failure cases must still resolve direct
commands through deterministic matching. No unit test waits in real time or contacts Vertex.

## Real-device DEBUG test

1. Install the debug APK: `./gradlew installDebug`.
2. Open **IVI AI 助理**, scroll to **DEBUG integrated command**, and select
   **Open accessibility settings**.
3. Enable **IVI AI 助理（無障礙）** and return to the target screen.
4. Put an AAOS launcher/settings page containing a visible clickable **設定** control underneath
   IVI AI 助理, reopen the assistant, enter **開設定**, and press **Run verified command**.
5. The assistant backgrounds itself. Wait for the target screen to change, then return to the
   assistant to inspect:
   - `Snapshot`: `source=UnderlyingTarget`, the underlying package/screen name, and element count;
   - `Resolved by`: `LanguageModel` with valid Vertex configuration, otherwise
     `DeterministicFallback`;
   - `Action`: the validated decision and dispatch acceptance;
   - `Event wait`: `ChangeObserved` or bounded `TimedOut`;
   - `Verification`: `Verified`, `NotVerified`, or `Inconclusive` with its fixed reason.
6. Treat the run as successful only when Verification is `Verified`. `accepted=true` alone is not
   acceptance evidence.

### Live LM debugger (no action)

Enter an utterance and press **Test LM only (no action)**. The assistant uses the same event-first
background/reveal and live snapshot, calls the LM-first decision layer, but invokes no accessibility
action. After returning to the assistant, inspect the snapshot source, final route, schema-validated
goal/confidence or clarification/no-match, attempt count, and fixed last issue such as
`NotConfigured`, `Authentication`, `Network`, or `InvalidOrUnsafeResponse`. Raw model output,
prompts, tokens, and headers are never shown.

The older **matching inspector** and **verification comparison** are deliberately labelled
`SCRIPTED FAKE`; they remain deterministic Phase-2/3 regression tools and are not the live path.

## Device-only evidence still required

JVM tests cannot prove that a particular AAOS image emits the expected accessibility events, that
its launcher exposes Settings as clickable text, or that the live node remains valid through
dispatch. Record the screen before/after and the five DEBUG result lines during device acceptance.
Live `LanguageModel` routing additionally requires a permitted Vertex project/model, short-lived
token, IAM role, quota, and network; without them the expected route is deterministic fallback.

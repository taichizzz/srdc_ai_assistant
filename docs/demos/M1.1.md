# M1.1 — Streaming Speech-to-Text Acceptance

## Goal

Verify that **IVI AI 助理** captures push-to-talk Taiwan Mandarin, streams 16 kHz
mono PCM to Google Cloud Speech-to-Text V1, displays replaceable interim text, and
commits final text to the main **Accumulated final transcript**.

The DEBUG comparison harness additionally sends a repeated test utterance through
V1 `latest_short`, V2 `chirp_2`, and V2 `chirp_3`, then displays and logs timing,
outcome, and transcript evidence. It never changes the production V1 selection.

M1.1 remains push-to-talk. It has no wake word, open microphone, TTS, screen reading,
decision logic, or accessibility action.

## What each environment can prove

| Check | Local unit/build environment | Real mic device + live credential |
| --- | --- | --- |
| Reducer, cancellation, timeout, Retry | Required and deterministic | Observable in UI |
| Typed input without permission/credential | Required | Optional |
| Credential-presence check | Required with in-memory fakes | Required with local API key |
| Real microphone capture and release | Not provable on the mic-less AAOS emulator | Required |
| Google accepts the key and returns Mandarin text | No real network call in tests | Required |
| V2 config ordering, models, mapping, failure, cancellation | In-process gRPC fake | Optional live confirmation |
| Chirp model/region availability and comparative accuracy | Not provable locally | Required |

## 1. Automated gate — no credential or device required

From the repository root, run:

```sh
./gradlew testDebugUnitTest assembleDebug lintDebug assembleRelease
```

Expected:

- All unit tests pass, including API-key-only, token-only, both-configured, and
  neither-configured local checks.
- A fake production stream that emits no first result reaches recoverable **Error**
  after the 15-second virtual watchdog; **Retry** returns it to **Idle**.
- A fake stream that never completes after **Stop** reaches recoverable **Error**
  after the 5-second virtual drain watchdog; **Retry** returns it to **Idle**.
- Interim text replaces the prior partial; a final appends once and clears partial.
- Starting a new production session cancels and joins the previous fake session.
- Debug and release APKs assemble; lint has no blocking issue.

The timeout tests use coroutine virtual time. They do not wait 15 or 5 real seconds.

## 2. API-key setup — DEBUG builds only

These steps require a human-owned GCP project with Cloud Speech-to-Text enabled,
active billing/quota, and device access to `speech.googleapis.com`.

1. Keep the machine's existing `sdk.dir` in root `local.properties`.
2. Add the company-provided plain API key:

   ```properties
   GCP_STT_API_KEY=AIza...your-restricted-key
   GCP_STT_ACCESS_TOKEN=
   GCP_STT_PROJECT_ID=your-project-id
   GCP_STT_LOCATION=asia-southeast1
   ```

3. Restrict the key in Google Cloud Console to the Speech-to-Text API and apply the
   organization-approved application restrictions where supported.
4. Never paste a service-account JSON, private key, or credential file into the
   property. Never commit `local.properties`; it is gitignored.
5. Rebuild and install because the key is injected into debug BuildConfig at build
   time:

   ```sh
   ./gradlew installDebug
   ```

6. Launch **IVI AI 助理** and press **Check cloud configuration**.
7. Confirm **Configured locally — Google acceptance not checked**. This proves only
   local presence; the live recognition procedure below proves remote acceptance.

Release BuildConfig always contains an empty development API key. If both API key
and OAuth token are locally present, API-key mode takes precedence and only one
credential is sent.

`GCP_STT_PROJECT_ID` and `GCP_STT_LOCATION` are required by the V2 inline recognizer.
The default debug location is `asia-southeast1`, chosen because current Google model
tables list Taiwan Mandarin for Chirp 2 and Chirp 3 there. Availability can change;
confirm the project/model/location combination before formal sign-off. V1 does not
depend on the V2 location.

## 3. Production speak-to-text — real device and live credential required

Use a physical Android/AAOS device, or a phone AVD whose microphone input is known
to work. The standard AAOS emulator cannot prove microphone intake.

1. Start filtered diagnostics without printing credential values:

   ```sh
   adb logcat -s MicRecorder CloudSttClient SttViewModel
   ```

2. Launch **IVI AI 助理** and press the normal **Start** button—not the DEBUG cloud
   test. Grant microphone permission if Android asks.
3. Confirm **Session status: Listening**.
4. Say exactly **「你好」**.
5. While speaking, confirm **Current partial transcript** shows live text and may be
   replaced as Google revises its hypothesis.
6. Press **Stop**. Confirm status moves through **Stopping**, the microphone logs
   `AudioRecord released`, and the session reaches **Completed**.
7. Confirm **「你好」** appears once in the main **Accumulated final transcript** and
   **Current partial transcript** is empty after the final result.
8. Press normal **Start** again, say exactly **「打開設定」**, and press **Stop**.
9. Confirm **「打開設定」** appends once below **「你好」**, with no overlapping
   microphone/RPC session and no cancellation error.

The UI presents **Stop** while a session is active, so overlapping Start replacement
is exercised deterministically by `startCancelsPriorProductionSessionWithoutVisibleError`.
The second live Start above verifies a fresh session begins after prior cleanup.

## 4. DEBUG V1 / Chirp 2 / Chirp 3 comparison — real device and live credential required

This is an evaluation harness, not production routing. Use the same device, network,
microphone position, speaker, wording, and speaking style for all three runs. Repeat
the exact utterance rather than trying to run three microphone streams at once.

1. Complete the project/key/location setup in section 2 and install a fresh debug APK.
2. Start the CSV-friendly metrics log:

   ```sh
   adb logcat -s SttMetricsCsv
   ```

3. In **DEBUG Cloud STT test**, select **V1 (latest_short)**.
4. Press **Cloud STT test (DEBUG)**, say exactly **「你好，請打開冷氣」**, then press
   **Stop Cloud STT test** immediately after speaking.
5. Wait for **API outcome** to leave `running`. Record the displayed partial/final
   text and the one `SttMetricsCsv` row.
6. Repeat steps 3–5 with **V2 Chirp 2**, then **V2 Chirp 3**, using the exact same
   utterance and test conditions.
7. Confirm the main **Accumulated final transcript** did not change. The selector and
   results belong only to the DEBUG harness; normal **Start** remains V1.

The CSV columns are:

```text
engine,model,firstTokenMs,finalMs,totalMs,outcome,transcript
```

The panel and row mean:

- **First Token Latency**: first microphone chunk requested by the selected client to
  its first non-blank interim result. Score 5 below 150 ms, 3 at 150–350 ms, and 1
  above 500 ms.
- **Final Sentence Latency**: Stop/end-of-audio to a final result received afterward.
  Score 5 below 350 ms, 3 at 350–750 ms, and 1 above 1200 ms. `N/A` means no final
  arrived after Stop.
- **Total latency**: stream start to the latest final result.
- **Outcome**: `success`, `auth`, `permission`, `quota`, `network`, `timeout`,
  `not_configured`, `no_final`, `cancelled`, or `unknown`. Use outcome and latency
  across repeated rows for the API connection rubric (5 ≈ 100% and below 300 ms;
  3 ≥ 90% and 1–2 s; 1 = connection timeout) and the separate ≥95% reliability tier.
- **Transcript**: committed final text, or the latest interim if no final arrived.

The app cannot compute WER, code-switching accuracy, acoustic-noise robustness,
network resilience, fallback latency, dynamic hotword support, or billing cost. Use
the transcript/timing/outcome rows as evidence while a human runs the required 3G,
approximately 200 ms latency, 15% packet-loss, 100 km/h, and 65–70 dB scenarios and
checks provider billing externally.

If V2 returns a recoverable `permission`/`unknown` result while V1 succeeds, confirm
the exact project ID, `asia-southeast1` availability, API enablement/IAM, and whether
that project currently exposes the selected Chirp model. Never paste a credential
into a bug report or log.

## 5. Typed-input fallback — no mic or credential required

1. Build with both `GCP_STT_API_KEY` and `GCP_STT_ACCESS_TOKEN` blank or absent.
2. Launch the app without granting microphone permission.
3. Enter exactly **「顯示」** in **Transcript text**.
4. Press **Submit transcript**.
5. Confirm **「顯示」** appears exactly once in **Accumulated final transcript**,
   partial text is empty, and status is **Completed**.

This uses the same final-result reducer as production STT; it is not a separate fake
transcript path.

## 6. Recovery acceptance

### Not configured — no mic required for the local check

1. Leave both credential properties blank/absent, rebuild, and launch the app.
2. Press **Check cloud configuration**.
3. Confirm **Local configuration: Not configured**, status **Error**, and the fixed
   message: **Cloud speech is not configured. Add an approved credential and retry.**
4. Press **Retry** and confirm status returns to **Idle**.
5. Confirm typed input from section 5 still works.

With microphone permission available, pressing normal **Start** in the same build
must reach the same recoverable not-configured outcome instead of hanging or crashing.

### Connection/drain timeout — deterministic unit acceptance

Run the automated gate in section 1. The two fake-client tests prove:

- No first STT result for 15 seconds cancels and joins capture/RPC, clears partial,
  and exposes **Speech recognition timed out. Please retry.**
- No stream completion within 5 seconds after **Stop** does the same.
- **Retry** returns either timeout state to **Idle**.
- Receiving an interim/final result cancels the first-response watchdog, so ongoing
  speech that is making recognition progress is not subject to a total-session timer.

These timeout paths are intentionally verified with virtual time rather than by
making a live Google connection fail nondeterministically.

### Permission recovery — device UI check

1. Deny microphone permission and confirm a recoverable rationale and **Retry**.
2. Permanently deny it and confirm **Open settings** is available.
3. Grant permission in Settings, return, and confirm the app becomes usable without
   automatically opening the microphone.

## M1.1 acceptance record

Record the following during the live sign-off:

- Device/AAOS build and Android version.
- Debug APK commit SHA.
- Credential mode used (**API key**, never its value).
- Whether exact utterances **「你好」** and **「打開設定」** produced interim and
  final text at correct pitch-independent timing.
- Whether Stop released the microphone and the second session started cleanly.
- Any GCP API/quota/device issue, without copying credential or Authorization data.

Repository tests cannot establish live GCP acceptance, real microphone quality,
device audio routing, quota availability, or vehicle-network reachability. Those are
the explicit real-device items in sections 3 and 4.

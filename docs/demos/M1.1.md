# M1.1 — Streaming Speech-to-Text Demo

## Purpose

Verify that the AAOS application captures Taiwan Mandarin, streams it to Google
Cloud Speech-to-Text, displays interim text, and preserves the final transcript.

> **App name:** the launcher label is now **「IVI AI 助理」** (internal codename `seeandsay`).
> Earlier **SeeAndSay** / **所見即可說** mentions and the `SeeAndSayApi29` AVD name below refer
> to the same app and are kept as dated historical records.

## Current implementation status

Phase 5 provides the real Google Cloud Speech-to-Text V1 streaming client and an
isolated DEBUG microphone-to-cloud smoke test. The smoke tool shows interim/final
results but deliberately does not change the production transcript; production
Start remains microphone-only until Phase 6. Permission-independent typed transcript
injection remains available with no token or microphone.

## Prerequisites

1. Install the debug APK on an API 29+ Android or AAOS device.
2. Ensure the device has network access and a working microphone.
3. An OAuth access token is optional for building and typed-input testing. A fresh
   token is required for the Phase 5 live cloud smoke test.
4. Never place a service-account JSON file in the project or APK.

## Phase 4 Google Cloud authentication-plumbing checkpoint

### External prerequisites owned by the human team

Before Phase 5 acceptance, confirm outside this repository that:

1. A GCP project ID has been selected.
2. Cloud Speech-to-Text is enabled and billing/quota are active for that project.
3. Development may use a short-lived token from `gcloud auth print-access-token`.
4. Production has an organization-approved token broker; it must not package a
   developer token, service-account JSON, refresh token, or private key.
5. The target device can reach `speech.googleapis.com`.

Phase 4 does not block when these are unknown and makes no call capable of
verifying API enablement, quota, token acceptance, or device connectivity.

### No-token path

1. Leave `GCP_STT_ACCESS_TOKEN` absent or blank in gitignored `local.properties`.
2. Run `./gradlew testDebugUnitTest assembleDebug lintDebug`.
3. Launch **「IVI AI 助理」** and confirm the screen remains usable with
   **Local configuration: Not checked**.
4. Press **Check cloud configuration**.
5. Confirm the status becomes **Not configured**, the session becomes **Error**,
   and the fixed recoverable message explains that cloud speech is not configured.
6. Press **Retry** and confirm the session returns to **Idle** without crashing.
7. Without granting microphone permission, submit exactly **「顯示」** through the
   typed-input field and confirm it appears in the final transcript exactly once.

### Development-token presence path

1. Confirm the selected project has Speech-to-Text and billing/quota enabled.
2. Copy `local.properties.example` to the gitignored `local.properties`, preserving
   the machine's correct `sdk.dir`.
3. Set `GCP_STT_PROJECT_ID` to the selected non-secret project ID.
4. Run `gcloud auth print-access-token` and paste its output only into
   `GCP_STT_ACCESS_TOKEN` in `local.properties`.
5. Rebuild and reinstall the debug APK because BuildConfig is generated at build
   time and the token normally expires after approximately one hour.
6. Press **Check cloud configuration** and confirm the status becomes
   **Configured locally — Google acceptance not checked** with no error.
7. Confirm no token or Authorization header appears anywhere on screen or in
   filtered application logs.

This check proves only that a non-blank local value reached the provider. Google
acceptance, expiry, quota attribution, and authentication rejection are tested only
by the separate Phase 5 live smoke procedure below.

Verified on 2026-07-15 without a token on the API 29 `SeeAndSayApi29` phone AVD:
the APK installed and cold-launched as **「IVI AI 助理」**, initially showed
**Not checked**, mapped the check to **Error / Not configured** with the fixed
message and visible Retry/typed-input controls, and returned to **Idle** after
Retry. The configured branch is covered with an in-memory non-secret unit value;
no live token/project/quota was available for Google acceptance testing.

### Release and secrets checks

1. Run `./gradlew assembleRelease` with or without local development values.
2. Confirm release BuildConfig generates empty `GCP_STT_ACCESS_TOKEN` and
   `GCP_STT_PROJECT_ID` fields; only debug builds may consume local development
   configuration.
3. Confirm `git check-ignore local.properties` reports the file as ignored.
4. Confirm `.gitignore` still excludes `*.json`, service-account JSON names,
   keystores, `local.properties`, and generated build outputs.
5. Before committing, inspect `git diff` and verify no credential, Authorization
   header, private key, or generated BuildConfig output is staged.

## Phase 5 Google Cloud streaming checkpoint

### Deterministic in-repository verification

1. Leave `GCP_STT_ACCESS_TOKEN` absent or blank; no live credential is required.
2. Run:
   `./gradlew testDebugUnitTest assembleDebug lintDebug assembleRelease`
3. Confirm all unit tests pass and both APK variants assemble.
4. Confirm the in-process gRPC tests cover config-first ordering, subsequent PCM
   audio, interim/final mapping, final confidence, recoverable status mapping, and
   collector cancellation. These tests use no DNS, TLS, Google endpoint, or token.
5. Launch the debug APK and confirm the normal **Start** control still captures only
   microphone audio and does not create a transcript. Phase 6 owns production wiring.
6. Without granting microphone permission, submit exactly **「顯示」** through typed
   input and confirm it still appears once in the production final transcript.

### No-token cloud-smoke recovery

1. Use a debug build with `GCP_STT_ACCESS_TOKEN` absent or blank.
2. Press **Cloud STT test (DEBUG)** and grant microphone permission if Android asks.
3. Confirm the session leaves its busy state and shows the fixed recoverable message:
   **Cloud speech is not configured. Add a short-lived OAuth token and retry.**
4. Confirm neither raw cloud-result field nor the production transcript changes.
5. Press **Retry** and confirm the session returns to **Idle** without a crash.

### Live token and real-device smoke test

> Use a physical Android device, or a phone AVD with verified microphone input. The
> AAOS emulator cannot prove microphone intake. The device must reach
> `speech.googleapis.com`, and the chosen project must have Speech-to-Text enabled
> with active billing/quota.

1. Preserve the correct `sdk.dir` in gitignored `local.properties`.
2. Set non-secret `GCP_STT_PROJECT_ID` to the enabled billing/quota project.
3. Immediately run `gcloud auth print-access-token` and place only its output in
   `GCP_STT_ACCESS_TOKEN`. Never add a service-account JSON or private key.
4. Rebuild and reinstall the debug APK; the development token is injected at build
   time and normally expires in about one hour.
5. In a terminal, filter diagnostics with:
   `adb logcat -s CloudSttClient SttViewModel MicRecorder`
6. Launch **「IVI AI 助理」**, press **Cloud STT test (DEBUG)**, and grant microphone
   permission. Confirm status reaches **Listening**.
7. Say exactly **「你好」**. While speaking, confirm **Cloud partial result** shows
   replaceable interim text. Pause and confirm **Cloud final result** contains
   **「你好」** and a latest final confidence value appears.
8. Continue by saying exactly **「打開設定」**. Confirm raw interim text updates and
   the committed cloud result appends **「打開設定」** once.
9. Confirm **Accumulated final transcript** remains unchanged: the test is isolated
   verification and must not enter the production reducer.
10. Press **Stop Cloud STT test**. Confirm status passes through **Stopping** and
    reaches **Completed**, `AudioRecord released` appears, and no later PCM chunks
    are logged for that session.
11. Start and stop the cloud test again. Confirm only one RPC/microphone session is
    active and the previous session is cancelled/released first.
12. Replace the local token with an expired/invalid value, rebuild, and repeat the
    cloud test. Confirm the UI shows the fixed invalid/expired-token recovery message
    without exposing the token or Authorization header.
13. Disable device networking and repeat. Confirm the UI shows the fixed unreachable
    recovery message and does not remain stuck in Connecting, Listening, or Stopping.

The live steps above were not executable in this repository environment because no
GCP token/project was configured and real device microphone/network access was not
available. Google token acceptance, live quota attribution, actual Taiwan-Mandarin
recognition, device TLS/DNS connectivity, and real cancellation/release therefore
remain human verification items.

## Phase 2 permission and typed-input checkpoint

### Typed input without microphone permission

1. Clear the application's data or start with microphone permission not requested.
2. Launch **「IVI AI 助理」**.
3. Confirm **Microphone permission: Not requested** and **Session status: Idle**.
4. Do not press **Start** and do not grant microphone permission.
5. Enter exactly **「顯示」** in **Transcript text**.
6. Press **Submit transcript**.
7. Confirm **「顯示」** appears exactly once under **Accumulated final transcript**.
8. Confirm the partial transcript is empty and the session status is **Completed**.

### Granted permission and push-to-talk state

1. Press **Start**.
2. Grant the microphone permission in Android's dialog.
3. Confirm **Microphone permission: Granted**, **Session status: Listening**, and the
   button label **Stop**.
4. Confirm no transcript text was fabricated by pressing Start.
5. Press **Stop**.
6. Confirm **Session status: Completed** and the button label returns to **Start**.

### Retryable denial

1. Reset/revoke microphone permission and press **Start**.
2. Press **Deny** in Android's permission dialog.
3. Confirm **Microphone permission: Denied — retry available** and
   **Session status: Error**.
4. Confirm the rationale text and **Retry** action are visible.
5. Press **Retry** and confirm the screen returns to **Idle** without crashing.
6. Press **Start** and confirm Android can present the permission request again.

### Permanent denial

1. In Android's repeated permission dialog, select **Deny & don’t ask again**.
2. Confirm **Microphone permission: Permanently denied — open settings** and
   **Session status: Error**.
3. Confirm **Retry** and **Open settings** are both visible.
4. Press **Open settings** and confirm Android opens the app's application-details page.
5. Grant microphone permission in Settings and return to the app.
6. Confirm permission changes to **Granted** and the session returns to a usable
   **Idle** state without starting automatically.

Verified on 2026-07-15 with an API 29 phone AVD. Typed input committed while
permission remained not requested; retryable denial, permanent denial, Settings
recovery, grant, Start, and Stop all completed without a crash. The automated ADB
input used ASCII text while the pure ViewModel test verified Traditional Chinese
text through the same reducer.

## Phase 3 microphone and raw-PCM checkpoint

> Use a real Android device or phone emulator/AVD whose microphone input and audio
> output are known to work. The AAOS emulator has no usable microphone and cannot
> prove this checkpoint.

### Production push-to-talk capture and logs

1. Launch the debug build of **「IVI AI 助理」** and grant microphone permission.
2. In a terminal, start filtered logs with:
   `adb logcat -s MicRecorder SttViewModel DebugAudioPlayer`.
3. Press **Start** and confirm **Session status: Listening**.
4. Speak continuously for 10–15 seconds, including exactly **「你好」** and
   **「打開設定」**.
5. Confirm logs repeatedly show `Captured PCM chunk #N: 3200 bytes`. An occasional
   documented short chunk is acceptable, but zero/negative chunks must become a
   visible error rather than a stuck screen.
6. Confirm the partial and final transcript areas remain unchanged; Phase 3 does
   not fabricate recognition results.
7. Press **Stop** and confirm the screen briefly moves through **Stopping**, then
   reaches **Completed**.
8. Confirm logs contain `AudioRecord released` and no additional capture chunks
   appear after Stop.
9. Immediately press **Start** again, speak briefly, and press **Stop**. Confirm a
   fresh chunk sequence starts and both sessions release without a crash.

### DEBUG Record & Play Back

1. In the debug build, locate **DEBUG audio verification**.
2. Confirm production **Start** is not active, then press
   **Record & Play Back (DEBUG)**.
3. Say exactly **「你好，這是麥克風測試」** for 2–4 seconds.
4. Confirm **Captured PCM** increases while recording and no audio plays yet.
5. Press **Stop & Play Back**.
6. Confirm capture reaches **Stopping**, the log prints `AudioRecord released`,
   and only then the UI shows **Playing captured PCM**.
7. Confirm the exact phrase is audible at natural pitch and speed, without obvious
   clipping or rapid/slow playback.
8. Confirm playback ends, microphone controls become enabled again, and neither
   partial nor final transcript text changes.
9. Repeat without manually stopping and confirm recording ends automatically near
   320,000 bytes/10 seconds, shows the cap message, releases the microphone, and
   begins playback.

### Cancellation and recovery

1. Start production capture, leave the app, then return and press **Start** again.
2. Confirm only one capture session owns the microphone and the prior session logs
   its release before the next session begins reading.
3. Revoke microphone permission in system Settings during or between sessions.
4. Confirm the active collector is cancelled, the UI returns to a recoverable
   permission state, and granting permission later does not auto-start recording.
5. Confirm typed input still submits exactly **「顯示」** with microphone permission
   denied or not requested.

The PCM math, ten-second cap, concatenation, prior-session cancellation, and
record-before-play ordering are covered by local unit tests. Audible quality,
actual microphone intake, device audio routing, and pitch/speed require the real
device check above and cannot be established by JVM tests or a mic-less AAOS AVD.

On 2026-07-15, the debug APK installed and cold-launched successfully on the API
29 `SeeAndSayApi29` phone AVD, and the Phase 3 controls were visible. That AVD
rejected AudioRecord construction as unsupported; SeeAndSay exited its busy state
and displayed the recoverable microphone failure without crashing or hanging.
Continuous chunks and audible loopback therefore remain pending on a physical
microphone/speaker device.

## Final M1.1 acceptance script

> Permission, controls, state UI, typed input, microphone capture, authentication,
> and the Google streaming client are implemented. Production transcript wiring is
> Phase 6; use the isolated DEBUG cloud test for Phase 5 recognition verification.

1. Launch **「IVI AI 助理」**.
2. Press **Start** and grant microphone permission when prompted.
3. Say exactly: **「你好」**.
4. While speaking, confirm a live partial transcript appears and is replaced as the
   recognizer revises it.
5. Press **Stop**.
6. Confirm the final transcript contains **「你好」** and the partial line clears.
7. Start a second session and say exactly: **「打開設定」**.
8. Confirm the final transcript contains **「打開設定」** exactly once.
9. Enter **「顯示」** in the typed-input debug field and submit it.
10. Confirm **「顯示」** enters the same final-transcript path without microphone use.

## Recovery checks

1. Deny microphone permission and confirm a recoverable permission message appears.
2. Disable networking during a session and confirm the UI leaves its busy state with a
   recoverable network message.
3. In the Phase 5 DEBUG cloud test, use an expired OAuth token and confirm
   authentication rejection is visible; the local configuration check still reports
   presence only and cannot detect expiry.
4. Stop a session and immediately start another; confirm the prior microphone and stream
   have been released.

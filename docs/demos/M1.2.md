# M1.2 — Text-to-Speech Final Acceptance

## Goal

Verify that **IVI AI 助理** speaks arbitrary Taiwan-Mandarin text through Google Cloud
Text-to-Speech, suspends until audible playback completes, stops cleanly, never records its own
speech, and falls back to Android's on-device Taiwan-Mandarin engine when cloud TTS is unavailable
or disabled.

M1.2 uses the unchanged provider-neutral contract:

```kotlin
interface TtsClient {
    suspend fun speak(text: String)
}
```

Cloud configuration is `cmn-TW`, voice `cmn-TW-Wavenet-A`, LINEAR16 WAV at 24 kHz, mono, PCM16.
Cloud playback parses the RIFF chunks and sends only the WAV `data` payload to AudioTrack.

## What is deterministic and what requires a device

| Acceptance item | JVM unit tests | Real device | Live credential/network |
| --- | --- | --- | --- |
| Idle → Speaking → Completed | Yes | UI confirmation | No |
| Typed cloud failure → recoverable Error | Yes | Optional | No |
| Stop cancels playback without Error | Yes, with fakes | Audible confirmation | Cloud Stop check requires it |
| New Speak cancels/replaces the prior utterance | Yes | Optional | No |
| Cloud failure → device fallback | Yes | Audible confirmation | No for missing-credential route |
| `CLOUD_TTS_ENABLED=false` → device only | Yes | Audible confirmation | No |
| WAV header/extra chunks stripped; malformed WAV rejected | Yes | Pitch/speed/noise confirmation | Cloud audio check requires it |
| Microphone controls blocked while Speaking | Yes | UI/mic confirmation | No |
| Google accepts the company credential and selected voice | No | Yes | Yes |
| Device engine has working `zh-TW` voice data | No | Yes | No |

## 1. Automated gate — deterministic, no network or Android audio

From the repository root:

```sh
./gradlew testDebugUnitTest assembleDebug lintDebug assembleRelease
```

Expected:

- All tests pass; debug and release APKs assemble; lint reports no blocking issue.
- `TtsViewModelTest` covers Idle → Speaking → Completed, typed cloud failure → recoverable Error,
  Stop → Idle without Error, retry, and cancellation/replacement of an in-flight utterance.
- `CloudTtsSynthesizerTest` covers request text/voice/format, response-byte mapping, API-key/bearer
  metadata exclusivity, missing configuration, recoverable provider status, and RPC cancellation.
- `CloudTtsClientTest` covers valid RIFF/WAVE parsing, an odd-sized padded extra chunk, playback of
  only PCM `data` bytes, malformed/short WAV recovery, and playback cancellation.
- `FallbackTtsClientTest` covers typed cloud failure → device fallback,
  `CLOUD_TTS_ENABLED=false` → device only, and cancellation without fallback.
- `DeviceTtsClientTest` covers asynchronous completion, engine error, cancellation/Stop, and missing
  Taiwan-Mandarin voice data through a fake device engine.
- `DebugAudioExclusionPolicyTest` covers TTS blocking microphone actions and active capture/draining
  blocking TTS start.

No unit test contacts Google, Android TextToSpeech, AudioTrack, a microphone, or a speaker.

## 2. DEBUG credential and flag setup

`local.properties` is gitignored. Never commit it, an API key, access token, Authorization header,
service-account JSON, or private key.

Use the approved company API key:

```properties
GCP_STT_API_KEY=your_restricted_company_api_key
GCP_STT_ACCESS_TOKEN=
CLOUD_TTS_ENABLED=true
```

The existing API key works for Cloud TTS and takes precedence over the token. It should be
restricted in Google Cloud to the approved APIs/application according to company policy.

Alternatively, leave the key blank and provide a short-lived OAuth token:

```properties
GCP_STT_API_KEY=
GCP_STT_ACCESS_TOKEN=your_short_lived_oauth_token
CLOUD_TTS_ENABLED=true
```

The Text-to-Speech API must be enabled and billing/quota active for the credential's project.
Rebuild and reinstall the debug APK after changing any BuildConfig-backed property:

```sh
./gradlew installDebug
```

## 3. Cloud-path acceptance — real device, live credential, and network required

Prerequisites:

- `CLOUD_TTS_ENABLED=true` and one approved credential configured as above.
- Device speaker/audio route available and media volume audible.
- Stable network access to `texttospeech.googleapis.com`.

Steps:

1. Launch the debug app and scroll to **DEBUG text-to-speech**.
2. Enter exactly **「你好」** and press **Speak**.
3. Confirm status changes to **Speaking** during synthesis/playback.
4. Confirm **「你好」** is audible in natural Taiwan Mandarin.
5. Confirm status changes to **Completed** only after the final audible sound.
6. Enter exactly **「已進入設定」** and press **Speak**.
7. Confirm the complete sentence is audible at natural pitch and speed.
8. Confirm there is no click/noise at the beginning from WAV header bytes and no unnaturally fast or
   slow playback from a 24 kHz sample-rate mismatch.

Correct pitch/speed plus clean audio provides the real-device evidence that the LINEAR16 WAV was
parsed and its PCM data played at 24 kHz. Unit tests provide the exact byte-level proof.

### Stop acceptance

1. Enter **「這是一段用來驗證停止播放的較長句子，請在句子結束以前按下停止。」**.
2. Press **Speak**, wait until audio begins, then press **Stop** before completion.
3. Confirm audio stops promptly, status becomes **Idle**, and no Error appears.
4. Confirm the utterance does not restart through device fallback after cancellation.

## 4. Device-fallback acceptance — real device required, cloud credential not required

Before testing, ensure the selected Android TTS engine has Taiwan-Mandarin (`zh-TW`) voice data.

### Missing-credential fallback

1. Set both `GCP_STT_API_KEY` and `GCP_STT_ACCESS_TOKEN` blank.
2. Keep `CLOUD_TTS_ENABLED=true`, rebuild, and reinstall.
3. Enter exactly **「你好」** and press **Speak**.
4. Confirm on-device Mandarin is audible, status reaches **Completed**, and no intermediate cloud
   Error is displayed.

### Flag-off device-only path

1. Set `CLOUD_TTS_ENABLED=false`, rebuild, and reinstall.
2. Enter exactly **「已進入設定」** and press **Speak**.
3. Confirm on-device Mandarin is audible and status reaches **Completed**.
4. This path must perform no cloud credential lookup or synthesis RPC; unit tests prove routing.

Restore `CLOUD_TTS_ENABLED=true` after the device-only check if cloud-first behavior is desired.

## 5. Echo-rule acceptance — real device/microphone required

1. Start a long TTS utterance and keep it in **Speaking**.
2. Confirm production STT Start, Cloud STT test, Record & Play Back, and all other microphone/audio
   entry controls are disabled throughout synthesis and playback.
3. Press **Stop** and confirm microphone controls become available only after TTS returns to Idle.
4. Start production STT, the cloud STT smoke test, or debug recording.
5. Confirm TTS **Speak** remains disabled through Connecting, Listening, and Stopping/draining.
6. Stop capture and confirm Speak becomes available only after microphone ownership is released.

The pure exclusion-policy tests verify these state gates. A real device verifies actual Android mic,
speaker, AudioTrack, TextToSpeech, and audio-focus behavior.

## 6. Acceptance record

Record the following for milestone sign-off:

- Git commit SHA and automated gate result/test count.
- Device/AAOS model, Android version, and audio route.
- Credential mode used: restricted API key or short-lived OAuth token—never record its value.
- Cloud results for **「你好」** and **「已進入設定」**.
- Whether pitch/speed were natural and startup audio contained no WAV-header noise.
- Whether Stop ended speech without Error or fallback restart.
- Device TTS engine/version and whether `zh-TW` data was installed.
- Missing-credential fallback and flag-off device-only results.
- Echo-rule control and real mic/speaker results.
